# 09. 확장 후보 제품화 기준 결정

작성 기준: 저장소 코드 정적 분석, 2026-03-17(KST)
범위: `P3-3 expansion candidate productization definition`

---

## 1. 목적

이 문서는 `/settings/data-sources`의 `확장 후보` 3개를
실제 제품 backlog 관점에서 어떻게 판정할지 먼저 고정하기 위한 결정 문서입니다.

이번 라운드의 목표는 아래 4가지입니다.

1. `retirement`, `insurance`, `macro`를 제품 카테고리와 host surface 기준으로 다시 분류한다.
2. 공통 판정 틀을 `사용자 효용 / 최신성·안정성 / 설명 가능성 / 운영 비용 / 규제·오해 리스크` 5축으로 고정한다.
3. 각 후보의 tier, 선행 조건, 공개 가능한 최소 surface, 보류 이유를 문서로 정리한다.
4. `P3-4`와 충돌하지 않는 가장 작은 rollout 순서를 정한다.

---

## 2. 현재 trust hub 기준 후보 자산

코드 기준 현재 후보는 아래 3개입니다.

| id | 현재 설명 | 현재 gate | 기반 source |
| --- | --- | --- | --- |
| `retirement` | 국민연금·퇴직연금·원리금보장 금리를 함께 보는 노후 준비 후보 | 데이터 계약과 비교 문구를 먼저 고정 | `NPS`, `FSC_RETIRE_PENSION`, `KDB_RETIRE_GUARANTEE_RATE`, `FSC_FIN_COMPANY_INFO` |
| `insurance` | 보험 정보를 연결해 보장 공백/중복 점검을 돕는 후보 | 보장 범위·면책 안내 문구 먼저 고정 | `FSS_INSURANCE` |
| `macro` | 금리·물가·고용 같은 거시지표를 플래닝 가정과 같이 보는 후보 | 국가·주기·단위·asOf 기준 먼저 고정 | `BOK_ECOS`, `KOSIS`, `FRED` |

해석:
- 현재 trust hub의 `확장 후보`는 backlog 후보를 보여 주는 카드이지, 공개 기능 상태를 의미하지 않습니다.
- 따라서 후보를 바로 route나 메뉴로 승격하지 않고, 먼저 `어디에 붙을 기능인지`와 `어떤 gate가 닫혀야 하는지`를 정하는 것이 맞습니다.

## 3. 공통 판정 틀

## 3.1 평가 축

후보는 아래 5축으로 읽습니다.

1. 사용자 효용
   지금 제품의 핵심 루프에 직접 도움이 되는가
2. 최신성·안정성
   source 상태와 시점 설명이 현재 구조에서 감당 가능한가
3. 설명 가능성
   비전문가에게 쉬운 한국어로 오해 없이 설명할 수 있는가
4. 운영 비용
   source 장애, fallback, 계약 관리 비용이 현재 팀 규모에서 감당 가능한가
5. 규제·오해 리스크
   보장 표현, 적합성 오해, 약관·정책 해석 리스크가 큰가

## 3.2 tier 정의

### Tier A — 바로 backlog 진입 가능

- 핵심 사용자 문제와 직접 연결됨
- 데이터 설명이 현재 구조에서 가능함
- 오해 리스크가 통제 가능함
- trust hub 후보 카드에서 실제 host surface backlog로 옮길 수 있음

### Tier B — 계약·문구·정책 선행 필요

- 사용자 효용은 있으나 바로 공개하기에는 gate가 남아 있음
- host surface는 정할 수 있지만 공개 조건을 먼저 문서/계약으로 닫아야 함
- backlog에 넣되, 구현 전 선행 설계 항목을 반드시 동반해야 함

### Tier C — 보류

- 현 시점 제품 문맥에서 오해 리스크 또는 운영 비용이 큼
- trust hub candidate로만 남기고 public feature backlog로는 올리지 않음
- 선행 정책/문구/계약이 없으면 제품화하지 않음

## 4. 후보별 판정

## 4.1 retirement

### 분류

- 제품 카테고리: 장기 재무결정 지원
- 권장 host surface: `/planning`
- 보조 연결 가능 surface: `/recommend`

### 5축 판정

| 축 | 판정 | 이유 |
| --- | --- | --- |
| 사용자 효용 | 높음 | 노후 준비는 플래닝과 직접 연결되는 장기 질문임 |
| 최신성·안정성 | 중간 | 공적 연금·퇴직연금·보장금리 source를 함께 읽어야 하고 asOf 설명이 중요함 |
| 설명 가능성 | 중간 | “예상 수령액”, “보장금리”, “노후 충분성” 표현이 과장되면 오해 위험이 큼 |
| 운영 비용 | 중간 | source 계약과 비교 기준을 먼저 맞춰야 함 |
| 규제·오해 리스크 | 높음 | 수익 보장·노후 충분성처럼 들리지 않게 문구를 강하게 통제해야 함 |

### 결정

- 추천 tier: `Tier B`
- 공개 가능한 최소 surface:
  - 1순위: `/planning`의 장기 준비 보조 블록
  - 2순위: `/recommend` 결과 설명의 장기 준비 참고 맥락
- 선행 조건:
  - 연금/퇴직연금/보장금리 비교 문구 고정
  - 기대값·보장값·가정값 구분 규칙 문서화
  - 기준 시점과 source별 asOf 노출 방식 합의
- 아직 보류해야 하는 이유:
  - 현재 구조로 바로 공개하면 “노후 보장”, “수익 확정”처럼 오해될 위험이 큼

## 4.2 insurance

### 분류

- 제품 카테고리: 보장 공백 점검
- 권장 host surface:
  - 없음. 현재 단계에서는 trust hub candidate 유지

### 5축 판정

| 축 | 판정 | 이유 |
| --- | --- | --- |
| 사용자 효용 | 중간 | 관심은 크지만 실제 질문은 약관·면책·보장 범위 해석에 좌우됨 |
| 최신성·안정성 | 중간 | source 자체보다 약관 맥락과 상품 세부조건 해석이 더 중요함 |
| 설명 가능성 | 낮음 | 비전문가에게 “보장 충분/부족”을 단순 문장으로 말하기 어려움 |
| 운영 비용 | 높음 | 약관·예외·면책·갱신형 차이를 설명하는 비용이 큼 |
| 규제·오해 리스크 | 높음 | 상품 권유 또는 보장 판단 대체로 읽힐 위험이 큼 |

### 결정

- 추천 tier: `Tier C`
- 공개 가능한 최소 surface:
  - 현재는 없음
  - trust hub의 `확장 후보` 카드 수준 유지
- 선행 조건:
  - 약관 판단을 대체하지 않는 copy 정책
  - 보장 범위·면책·갱신/비갱신 안내 규칙
  - 오해를 줄이는 결과 구조와 책임 문구
- 아직 보류해야 하는 이유:
  - 지금 제품의 설명 구조와 신뢰 레이어로는 보험 해석 리스크를 안전하게 감당하기 어려움

## 4.3 macro

### 분류

- 제품 카테고리: planning support layer
- 권장 host surface: `/planning`
- trust hub 잔류 이유: source 안내와 기준일 설명이 먼저 필요함

### 5축 판정

| 축 | 판정 | 이유 |
| --- | --- | --- |
| 사용자 효용 | 중간 | 직접 행동 기능보다 플래닝 가정 이해를 돕는 보조 가치가 큼 |
| 최신성·안정성 | 중간 | 국가·주기·단위가 다른 시계열을 같이 읽어야 해 asOf 규칙이 중요함 |
| 설명 가능성 | 중간 | 지표 자체는 낯설지만 planning 가정 보조로 좁히면 설명 가능함 |
| 운영 비용 | 중간 | source는 많지만 support layer로 제한하면 범위를 관리할 수 있음 |
| 규제·오해 리스크 | 중간 | 거시지표를 투자 판단처럼 읽히지 않게 제어해야 함 |

### 결정

- 추천 tier: `Tier B`
- 공개 가능한 최소 surface:
  - 1순위: `/planning` 가정/해석 보조 블록
  - 2순위: planning report의 기준 설명 보조
- 선행 조건:
  - 국가·주기·단위·asOf 표기 규칙 고정
  - “참고 지표”와 “행동 권고”의 경계 문구 고정
  - P3-4에서 환율/공시/혜택/주거와의 역할 분리 기준 연결
- 아직 보류해야 하는 이유:
  - 직접 public feature로 열면 해석 난도가 커지고, 현재 제품의 행동 루프보다 설명 부담이 앞설 수 있음

## 5. rollout 우선순위

`[권장안]`

1. `macro`
   - 이유: 독립 기능이 아니라 `/planning` support layer로 가장 작게 붙일 수 있고, 보험보다 규제 리스크가 낮음
2. `retirement`
   - 이유: 사용자 효용은 높지만 비교 문구와 가정 구분 정책을 먼저 닫아야 함
3. `insurance`
   - 이유: trust hub candidate 유지, public backlog 보류

보완 원칙:
- 후보를 trust hub 카드에서 바로 내보내지 않습니다.
- host surface가 없는 후보는 backlog-ready가 아닙니다.
- `P3-4`가 정할 역할 분리 기준을 통과하기 전에는 독립 메뉴나 독립 public route를 만들지 않습니다.

---

## 6. P3-4와의 연결 기준

`P3-3`는 후보 자체를 줄 세우는 단계이고,
`P3-4`는 실제 공개될 때 어느 행동 문맥에 붙을지를 고정하는 단계입니다.

연결 원칙:
- `macro`는 `planning` 가정 보조 레이어로 들어갑니다.
- `retirement`는 `planning` 또는 `recommend`의 장기 준비 보조 문맥에서만 검토합니다.
- `insurance`는 역할 분리보다 먼저 policy/copy gate가 닫혀야 하므로 `P3-4` 이전 단계에서 계속 보류 가능합니다.

즉, 후보는 trust hub에서 “준비 상태”를 보여 줄 수 있지만,
실제 공개는 `planning / recommend / public info`의 host surface가 정해진 뒤에만 가능합니다.

---

## 7. 이번 단계 결론

- `P3-3`는 구현보다 backlog 기준 고정이 먼저입니다.
- 현재 기준에서 `Tier A` 후보는 없습니다.
- `macro`, `retirement`는 `Tier B`, `insurance`는 `Tier C`로 두는 것이 가장 안전합니다.
- 다음 구현 착수 후보는 `macro`의 planning support layer 설계이며,
  `insurance`는 trust hub candidate로만 유지하는 것이 맞습니다.
