# 01. 현행분석 및 개선기획서

## 1. 문서 목적

`financeproject`의 현재 공개 저장소를 기준으로 제품 구조를 파악하고,
향후 유지보수성과 사용자 이해도, 배포 안정성을 높이기 위한 개선 방향을 제안합니다.

## 2. 현행 제품 정의

### 2.1 제품 한 줄 정의

[현행 확인]
이 저장소는 Next.js 기반의 금융 워크스페이스이며, README 기준으로 개인 재무설계, 금융상품 추천, 공공데이터 연동, DART 공시 모니터링을 하나의 서비스 안에 통합하고 있습니다.

### 2.2 현재 포함된 주요 도메인

| 도메인 | 대표 경로 | 성격 | 비고 |
|---|---|---|---|
| 재무설계 v2 | `/planning`, `/planning/runs`, `/planning/reports`, `/planning/trash` | 핵심 사용자 기능 | 문서와 테스트가 가장 잘 정리됨 |
| 재무설계 v3 | `/planning/v3/*` | 실험/확장 트랙 | v2 freeze와 병행 개발 |
| 상품 추천 | `/recommend`, `/recommend/history` | 사용자 기능 | planningContext 연결 흔적 존재 |
| 상품 탐색/비교 | `/products`, `/products/catalog`, `/products/compare` | 사용자 기능 | unified catalog 기반 |
| DART/공시 | `/public/dart`, `/public/dart/company`, `/invest/companies` | 사용자 기능 + 모니터링 | 검색/모니터/인덱스 안내가 함께 있는 복합 허브 |
| 대시보드 | `/dashboard` | 통합 진입점 | 사용자 정보와 운영 상태가 함께 노출됨 |
| 설정/데이터소스 | `/settings/*`, `/settings/data-sources` | 사용자/운영 경계 기능 | 운영 최신 기준, 연결 테스트 혼재 |
| 운영도구 | `/ops/*` | local-only 운영 기능 | assumptions, security, cache, audit 등 |
| 개발/디버그 | `/debug/*`, `/dev/*`, `/api/dev/*` | 개발자 기능 | production 404 정책 존재 |

### 2.3 아키텍처/저장 구조의 현행 특징

[현행 확인]
- 프론트엔드/서버는 Next.js App Router 중심입니다.
- 제품/외부상품/즐겨찾기 계열은 Prisma + SQLite 모델이 존재합니다.
- Planning v2/v3, assumptions, backup, doctor, news 등은 `.data/**` 기반의 local-first 흐름이 강합니다.
- Planning v2 문서는 핵심 계산 엔진을 pure/offline 영역으로 두고, 네트워크/운영 경계만 server/ops에서 처리하도록 명시합니다.

[해석]
즉, 이 저장소는 단일 저장 전략이 아니라 **DB 기반 모델 + 파일 기반 로컬 저장소**가 공존하는 구조입니다.
현재 단계에서는 빠른 실험에 유리하지만, 장기적으로는 DTO 정합성·백업·권한·멀티유저 확장 시 복잡도가 크게 올라갑니다.

## 3. 현행 강점

### 3.1 문서화 수준이 높음

[현행 확인]
- `docs/current-screens.md`로 현재 화면 카탈로그가 정리되어 있습니다.
- `docs/routes-inventory.md`로 공개/운영/디버그 URL과 내부 API가 구분되어 있습니다.
- `docs/planning-v2-onepage.md`, `docs/planning-v2-user.md`, `docs/planning-v2-architecture.md`가 Planning v2의 사용자 흐름과 아키텍처 기준을 설명합니다.
- `docs/frontend-design-spec.md`가 UI 상태와 디자인 일관성 기준을 규정합니다.

[해석]
기능이 많아도 문서와 가드가 따라오는 구조는 큰 장점입니다. 이 프로젝트는 “무질서하게 커진 실험 저장소”라기보다, **운영 가능성을 의식하면서 확장 중인 저장소**에 가깝습니다.

### 3.2 v2 안정화 / v3 확장 분리 의식이 있음

[현행 확인]
- 백로그 문서에서 v2는 stabilization only, 신규 가치 증가는 v3에서 진행하도록 명시합니다.
- v3 kickoff 문서도 별도 트랙으로 정의되어 있습니다.

[해석]
이는 장기적으로 올바른 방향입니다. 다만 문서 정책과 실제 IA가 완전히 분리되지 않아 사용자가 체감하는 경계는 아직 약합니다.

### 3.3 운영 보안/가드 의식이 강함

[현행 확인]
- planning/ops write API에 same-origin/CSRF/confirmText 검증이 반복적으로 구현되어 있습니다.
- debug/dev 경로는 production에서 차단되도록 테스트와 문서가 맞물려 있습니다.
- 로컬 전용, packaged runtime, production 비노출 정책이 분명합니다.

[해석]
이 프로젝트는 기능보다 guardrail을 먼저 세우는 문화가 일부 정착되어 있습니다. 운영 실수 방지에는 매우 유리합니다.

### 3.4 자동화 테스트 자산이 이미 존재함

[현행 확인]
- `tests/e2e`에는 planning fast/full/quickstart, history/report, data-sources, debug access, DART search/company/monitor/build, v3 import/draft apply, news rules, smoke 등 주요 회귀 경로가 존재합니다.

[해석]
새 문서를 만드는 것보다, **기존 테스트를 요구사항 문서와 연결**하는 것이 비용 대비 효과가 큽니다.

## 4. 핵심 문제 진단

## 4.1 문제 1 — 제품 경계가 사용자 눈높이에서 불명확함

[현행 확인]
- 공개 경로만 해도 dashboard, planning, recommend, products, DART, public APIs, settings가 넓게 펼쳐져 있습니다.
- 동시에 planning v3 실험 화면이 같은 제품 트리 안에 존재합니다.
- ops/debug/dev 경로도 동일 저장소와 문서 체계 안에 공존합니다.

[해석]
현재 저장소의 가장 큰 문제는 기능 부족이 아니라 **경계의 불투명성**입니다.
사용자는 “이 서비스의 핵심이 무엇인지”를 바로 이해하기 어렵고, 팀 내부에서도 “운영 화면인지, 사용자 화면인지, 실험 화면인지”가 섞여 보일 수 있습니다.

[영향]
- 신규 사용자 온보딩 비용 증가
- 메뉴/홈/대시보드의 정보 설계 난이도 상승
- QA 범위 확장
- 배포 시 노출 정책 실수 가능성 증가

### 4.2 문제 2 — 대시보드가 사용자 랜딩과 운영 스냅샷 역할을 동시에 수행함

[현행 확인]
`/dashboard`는 최근 planning run 외에도 data source status, source freshness, FX preview, benefits, feedback 최근 항목 등을 함께 로딩합니다.

[해석]
현재 대시보드는 “개인 금융 홈”이 아니라 **개인 금융 + 운영 상태 + 외부 정보 모니터**가 섞인 혼합 허브에 가깝습니다.
이 방식은 내부 사용자에게는 편리할 수 있으나, 일반 사용자에게는 핵심 CTA가 흐려집니다.

[권장]
- 사용자용 Home Dashboard와 운영용 Status View를 분리합니다.
- 사용자 홈은 “내 계획 / 내 추천 / 내 기록 / 다음 액션” 중심으로 축소합니다.
- 운영/데이터 freshness는 `/settings/data-sources` 또는 `/ops` 성격으로 이동합니다.

### 4.3 문제 3 — v2 freeze 정책은 명확하지만, 제품 IA는 아직 분리되지 않음

[현행 확인]
- 문서는 v2 안정화, v3 신규 확장 정책을 명시합니다.
- 하지만 route tree 상에서는 모두 `/planning` 하위에 존재합니다.

[해석]
개발 정책과 사용자 IA가 다르면, 결국 문서만 분리되고 경험은 분리되지 않습니다.
지금 구조는 내부적으로는 “분리 개발”이지만, 외부에서는 “planning이 왜 이렇게 많지?”로 보일 가능성이 큽니다.

[권장]
- v3는 최소한 아래 3가지 중 하나로 명시적으로 분리해야 합니다.
  1. `Labs`/`Beta` 뱃지
  2. 메뉴 비노출 + deep link only
  3. 별도 feature flag/entry point

### 4.4 문제 4 — 데이터 저장 전략이 이원화되어 DTO 표준화가 어렵다

[현행 확인]
- Prisma schema에는 product/provider/favorite/external snapshot/user profile 성격의 모델이 있습니다.
- planning 문서와 v3 user guide는 `.data/**` local-first 저장을 강조합니다.
- planning v2 store type은 profile/run/report와 재현성 정보, actionCenter, blobRef 등을 파일 저장 친화적으로 정의합니다.

[해석]
현재는 **“운영 DB 모델”과 “로컬 워크스페이스 모델”이 분리된 상태**입니다.
문제가 되는 지점은 저장소가 둘이라는 사실 자체가 아니라, 이 둘 사이를 잇는 canonical DTO가 아직 제품 차원에서 명문화되지 않았다는 점입니다.

[영향]
- profile/runs/recommendation 간 데이터 재사용 난이도 증가
- backup/restore 범위 혼동
- multi-user/encryption 확장 시 마이그레이션 복잡도 증가
- 화면마다 서로 다른 필드 해석 발생 위험

### 4.5 문제 5 — planning과 recommend의 연결은 시작됐지만 아직 제품 경험으로 완성되지 않음

[현행 확인]
- 추천 요청 DTO에 `planningContext`가 존재합니다.
- 추천 API는 planning linkage/readiness meta를 계산하지만 stage inference는 disabled로 남아 있습니다.

[해석]
현재 구조는 “계획 수립”과 “상품 추천”이 기술적으로 이어질 준비는 돼 있지만, **사용자 경험으로는 아직 분리된 두 기능**에 가깝습니다.
이 상태가 길어지면 recommend는 독립 필터 UI로 남고, planning은 결과 소비가 약한 계산기처럼 보일 수 있습니다.

[권장]
- `Planning Run -> Recommendation Profile` 자동 생성 규칙을 확정합니다.
- 사용자가 planning 결과에서 바로 추천으로 이동할 때, 금액/기간/유동성/부채 상태가 자동 반영되게 합니다.
- 추천 결과에서 “왜 이 상품이 내 계획과 맞는지” 설명 문장을 표준화합니다.

### 4.6 문제 6 — API 응답 도우미 체계가 일부 이질적일 가능성이 높다

[현행 확인]
- planning simulate route는 `ok/fail` helper를 사용합니다.
- planning profile/trash 계열은 `jsonOk/jsonError` helper를 사용합니다.

[해석]
내부 구현 의도는 비슷해도, 응답 envelope가 완전히 동일하다고 보장하기 어렵습니다.
이런 미세한 차이는 프론트엔드 에러 처리, QA, API 문서 자동화에서 비용을 만듭니다.

[권장]
- `ApiSuccess<T>`, `ApiError`, `ApiMeta` 규약을 전 구간에 통일합니다.
- 특히 `meta`, `issues`, `code`, `status`, `generatedAt`, `cache` 필드의 위치를 고정합니다.

### 4.7 문제 7 — `/settings/data-sources`의 소유권이 사용자 화면과 운영 화면 사이에서 애매함

[현행 확인]
- 해당 페이지는 production에서는 read-only “운영 최신 기준” 중심으로 보이고, 비프로덕션에서는 ping/diagnostics 기능이 더 드러납니다.
- e2e도 “운영 최신 기준” 블록과 연결 테스트 버튼 반영을 함께 검증합니다.

[해석]
이 페이지는 이름은 settings지만 실제 역할은 **사용자 설정**보다 **운영 데이터 상태 허브**에 가깝습니다.

[권장]
- 사용자용 `설정`과 운영용 `데이터 상태`를 분리합니다.
- 사용자 설정은 alerts/backup/recovery 등 개인 작업만 남깁니다.
- 데이터 freshness와 ping은 `/ops` 또는 `/admin/status` 성격으로 이동하는 것이 명확합니다.

### 4.8 문제 8 — DART 화면은 기능이 있지만 첫 진입 설명이 아직 복합적이다

[현행 확인]
- `/public/dart`는 `DartSearchClient`를 사용하며 검색 탭, monitor 탭, 최근 검색, 즐겨찾기, missing index 복구 안내를 함께 제공합니다.
- 회사 상세(`/public/dart/company`)는 기업 기본 정보와 monitor 진입 액션을 제공하고, monitor 탭은 즐겨찾기 기반 우선순위 보기를 지원합니다.

[해석]
실기능은 이미 검색/모니터 제품 수준까지 올라와 있지만, 첫 화면에서 검색, 인덱스 준비 상태, monitor 진입 맥락이 함께 보여 초기 이해 비용이 아직 있습니다.

[권장]
- `/public/dart` 첫 화면에서 검색 시작, monitor 진입, 인덱스 복구 안내의 위계를 더 명확히 나눕니다.
- 정식 승격 시 `/public/dart` 첫 화면에 검색/관심기업/오늘 브리프 중 하나를 명확한 primary CTA로 둡니다.

## 6. 핵심 개선 방향

### 6.1 방향 A — “계획 → 추천 → 실행” 사용자 여정을 1개의 골격으로 묶기

[권장]
현재 가장 높은 제품 가치가 나오는 축은 다음 3단계입니다.

1. 사용자가 프로필 입력/수정
2. planning run 수행 및 결과 확인
3. 결과를 추천상품/액션으로 연결

따라서 대시보드와 메뉴도 이 3단계를 우선 배치해야 합니다.

### 6.2 방향 B — Canonical Financial Profile 계약을 확정하기

[권장]
현재 v2 profile, v3 exposure/journal/account/batch 흐름, recommend planningContext가 분산돼 있습니다.
이들을 묶는 공통 계약을 아래처럼 두는 것이 좋습니다.

- 개인 기본 정보
- 월간 현금흐름
- 자산/부채
- 목표
- 위험성향/유동성 성향
- 정책/가정치 참조
- 출처(source of truth) 및 최신 동기화 시간

이 계약이 정해져야 screen, API, backup, QA가 모두 단순해집니다.

### 6.3 방향 C — 저장 전략은 통일보다 “추상화”가 먼저다

[권장]
즉시 DB 하나로 통합하기보다, 먼저 저장 추상화를 명확히 하는 것이 현실적입니다.

- `ProfileRepository`
- `PlanningRunRepository`
- `ArtifactRepository`
- `CatalogRepository`
- `WorkspaceBackupRepository`

이 추상화가 먼저 있어야 v2/v3 병행과 local-first 전략을 유지하면서도 기술 부채를 줄일 수 있습니다.

### 6.4 방향 D — 화면 상태 정의를 공통 DoD로 강제하기

[현행 확인]
frontend design spec은 Loading/Empty/Error/Disabled 상태 분리를 요구합니다.

[권장]
모든 핵심 화면 정의서와 QA 명세서에 아래를 필수로 넣어야 합니다.

- 로딩 표시 방식
- 빈 상태 문구 및 기본 CTA
- 오류 메시지와 재시도
- 비활성 조건과 사유
- 부분 성공(partial success) 처리 방식

### 6.5 방향 E — 릴리즈 레인 분리

| 레인 | 목적 | 허용 변경 |
|---|---|---|
| Stable | v2 및 현재 public UX 안정화 | 버그 수정, 문구 정리, 상태 개선, QA 강화 |
| Growth | recommend/products/DART 경험 강화 | 사용자 가치 증가 기능 |
| Labs | v3 실험 및 데이터 모델 검증 | 실험 기능, feature flag 기반 |
| Ops | 운영도구/보안/백업/진단 | local-only/ops 기능 |

## 7. 우선순위 로드맵

### 7.1 P0 (가장 먼저)

1. Public/ops/labs/dev 노출 정책 문서와 메뉴를 완전히 일치시킵니다.
2. `/dashboard`를 사용자 홈 기준으로 축소하거나, 운영 상태 섹션을 별도 화면으로 분리합니다.
3. recommend와 planning 사이의 canonical input/output 계약을 확정합니다.
4. API 성공/오류 envelope를 통일합니다.
5. `/settings/data-sources`의 소유권을 사용자용/운영용으로 재분류합니다.
6. DART 랜딩 문구와 실제 기능 상태를 일치시킵니다.

### 7.2 P1

1. Planning v3를 실험실 모듈로 명확히 표시합니다.
2. local-first 저장소와 DB 모델 사이에 repository adapter를 둡니다.
3. 추천 결과에 “내 계획과의 적합성 근거”를 표준 문장으로 노출합니다.
4. 리포트/실행기록/액션센터를 하나의 “Decision History” 관점으로 묶습니다.

### 7.3 P2

1. multi-user / encryption / shared workspace 전략 확정
2. tax/pension/optimizer formalization
3. 운영 KPI dashboard와 dry-run report 강화
4. 문서-테스트-화면 카탈로그의 자동 동기화 수준 상향
