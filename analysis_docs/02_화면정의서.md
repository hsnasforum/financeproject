# 02. 화면정의서

## 1. 문서 목적

이 문서는 `financeproject`의 현행 화면을 사용자/운영/실험/개발 관점으로 구분하고,
실제 운영 문서로 사용할 수 있는 수준의 화면 목적, 주요 기능, API 연계, 상태 정의, 개선 포인트를 정리합니다.

---

## 2. 화면 분류 기준

| 구분 | 의미 | 노출 원칙 |
|---|---|---|
| Public | 일반 사용자에게 노출되는 기능 | 메뉴/홈/직접 접근 허용 |
| Legacy / Redirect | 기존 진입점을 신경 써야 하는 이전 경로 | 공식 경로로 안전하게 이동 |
| Prototype / Preview | 기본 진입은 막고 preview query가 있을 때만 보여주는 미리보기 화면 | 기본은 공식 경로로 redirect |
| Planning v3 Experimental | 실험 기능 | 배지/비노출/feature flag 중 하나 필요 |
| Ops | 운영자 기능 | dev 기본 허용, production은 packaged runtime에서만 허용 |
| Dev / Debug | 개발자 기능 | non-production + debug flag + local request에서만 허용 |

## SCR-01. 홈 (`/`)

| 항목 | 정의 |
|---|---|
| 목적 | 최근 플랜, 오늘 볼 액션, 핵심 서비스로 이어주는 홈 허브 |
| 사용자 | 신규/기존 사용자 |
| 주요 기능 | 최근 플랜 이어보기, 첫 플랜 시작, 혜택/상품/화면 바로가기, 데이터 소스 준비 상태 요약 |

## SCR-03. 재무설계 워크스페이스 (`/planning`)

| 항목 | 정의 |
|---|---|
| 목적 | 프로필 선택/편집 후 planning 실행 |
| 사용자 | 핵심 사용자 |
| 주요 영역 | 프로필 선택, quickstart/wizard, snapshot 선택, 실행 옵션, 실행 버튼 |
| 연계 API | `/api/planning/v2/profiles`, `/api/planning/v2/runs`, `/api/planning/v2/simulate` 계열 |
| 주요 입력 | profile, horizonMonths, policyId, snapshotId, assumptions override |
| 주요 출력 | 시뮬레이션/시나리오/몬테카를로/액션/부채 전략 결과 진입 |
| 상태 정의 | Loading: 프로필/스냅샷 목록 로딩 / Empty: 프로필 없음 / Error: 입력 오류, snapshot 없음 / Disabled: health critical ack 전 저장 제한 등 |
| 접근 정책 | public |
| 개선 포인트 | 사용자가 이해할 수 있는 “입력 → 실행 → 추천” 흐름을 더 강하게 보여줘야 함 |

현행 UX 특징
- 선택 프로필과 baseline run 일부를 localStorage와 함께 다루는 흔적이 있습니다.
- v2 사용자 문서는 health critical ack가 있을 때 save/high-cost action을 막고 실행은 허용하는 정책을 설명합니다.

## 5. 공통 상태 정의 표준

`docs/frontend-design-spec.md` 기준으로 모든 핵심 화면에 아래 상태를 분리해서 구현하는 것을 기본으로 합니다.

| 상태 | 정의 | 기본 처리 원칙 |
|---|---|---|
| Loading | 데이터 로딩 중 | 레이아웃 흔들림이 적은 skeleton/placeholder 사용 |
| Empty | 데이터 없음 | 이유 설명 + 기본 CTA 1개 |
| Error | 요청/처리 실패 | 안전한 안내 + 다시 시도, 기존 입력 유지 |
| Disabled | 실행 불가 상태 | 숨기지 말고 비활성 사유를 근처에 명시 |
| Partial Success | 일부 단계만 성공 | 성공 데이터는 보여주고 실패 단계 이유를 별도 표시 |

## 6. 화면 설계 원칙

1. 모바일에서 테이블 사용을 지양하고 카드/리스트 중심으로 전환
2. 검색/필터/결과의 계층을 동일하게 유지
3. 사용자가 보는 public 화면과 운영/실험 화면의 용어를 구분
4. 리포트/실행기록/추천/상품탐색을 하나의 의사결정 흐름으로 연결
5. 모든 위험 액션(delete/purge/reset)은 confirmText 또는 동등한 보호장치 필요
