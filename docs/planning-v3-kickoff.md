# Planning v3 Kickoff

## A) 목적 / 범위
- v3 목적: v2 동결을 유지한 상태에서 신규 가치(연동/정밀 계산/최적화/멀티유저)를 별도 트랙으로 개발합니다.
- v3 포함 범위:
  - 계좌연동/마이데이터 계층 설계
  - 정밀 세금/연금 계산 모듈화
  - optimizer 정식 기능화(실험 플래그 해제 대상)
  - 멀티유저/권한/암호화 옵션 설계
- v3 제외 범위:
  - v2 코어 UX 대수정
  - v2 회귀 베이스라인 재정의
  - 단기 문구/라벨 수준 hotfix (v2 bugfix 트랙으로 처리)

## B) 우선순위 (Epic)

### Epic 1: 계좌연동/데이터 수집 경계 수립
- 목표/성공기준(측정):
  - 연동 소스별 ingestion 계약 정의 문서 확정
  - 로컬 저장 경계/마스킹 규칙 적용 확인
  - 샘플 데이터로 파이프라인 smoke PASS
- 리스크/의존성:
  - 외부 API 변동성, 인증/토큰 만료
  - 개인정보 경계 정책 선행 필요
- v2 영향:
  - 원칙적으로 없음 (v2 API/DTO 계약 불변)

### Epic 2: 정밀 세금·연금 계산 엔진
- 목표/성공기준(측정):
  - 단순 가정식 대비 편차 기준 정의 및 검증 케이스 확보
  - 계산 모듈 순수함수 테스트 PASS
  - 리포트용 요약 지표 일관 출력
- 리스크/의존성:
  - 제도 변경 반영 주기
  - 입력 모델 복잡도 증가
- v2 영향:
  - 원칙적으로 없음 (v3 전용 계산 경로)

### Epic 3: 최적화 정식화 + 멀티유저 기반
- 목표/성공기준(측정):
  - optimizer를 실험 단계에서 승격 가능한 품질 기준 수립
  - 멀티유저 저장/권한 모델 초안 및 위협 모델 문서화
  - v3 acceptance 시나리오 정의
- 리스크/의존성:
  - 연산 비용 증가, 데이터 격리 정책
  - 마이그레이션 전략 합의 필요
- v2 영향:
  - 원칙적으로 없음 (v2 run/profile 포맷 유지)

## C) 아키텍처 원칙
- 엔진 순수함수 원칙은 유지하고, I/O(연동/저장)는 경계 계층에서만 처리합니다.
- 네트워크/개인정보 처리는 서버 전용 계층으로 제한하고 토큰/원문 데이터 로그 금지 원칙을 유지합니다.
- 저장 포맷 변경은 마이그레이션 정책(P97-33)을 따르며, v2 데이터는 역호환을 우선합니다.

## D) 운영 / 품질
- v3에서도 `complete/regress/acceptance` 게이트 체계를 유지합니다.
- v2 regression baseline은 수정하지 않습니다.
- v3 신규 작업은 v3 문서/브랜치/모듈로 분리하고, v2에는 bugfix/경미 개선만 반영합니다.

## 라벨 정책
- v2: `v2-bug`, `v2-improve`
- v3: `v3-epic`, `v3-feature`, `v3-research`
