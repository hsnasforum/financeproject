# Project UI / Frontend Direction

## Hard Override
기존 디자인은 보호 대상이 아니다.
현재 UI가 일관되지 않거나, 올드하거나, 프로젝트 분위기와 맞지 않으면 반드시 수정한다.
부분 보수식 접근을 하지 말고, 필요한 범위에서 구조적으로 통일한다.
"이미 있는 디자인"보다 "더 나은 일관성"을 우선한다.

## Mission
이 프로젝트의 프론트엔드는 프로젝트의 성격과 사용자 기대에 맞는, 현대적이고 완성도 높은 경험을 제공해야 한다.

현재 존재하는 UI는 참고 자료일 뿐, 반드시 유지해야 하는 기준이 아니다.
기존 디자인이 일관되지 않거나, 낡아 보이거나, 제품 분위기와 맞지 않으면 적극적으로 수정하고 재설계한다.

## Product Mood
이 프로젝트가 주어야 하는 인상:
- [예: 신뢰감 있음]
- [예: 세련됨]
- [예: 가볍지만 전문적임]
- [예: 빠르고 명확함]

피해야 하는 인상:
- 올드함
- 조잡함
- 템플릿 느낌
- 중구난방 느낌
- 화면마다 다른 제품처럼 보이는 상태

브랜드/무드 정보가 명확하지 않다면, 제품의 도메인, 핵심 사용자, 콘텐츠 성격을 보고 적절한 방향을 추론한 뒤 끝까지 일관되게 유지한다.

## Priority Order
의사결정 우선순위는 아래와 같다.
1. 프로젝트 분위기와 브랜드 적합성
2. 전체 UI 일관성
3. 사용성 및 정보 위계
4. 현대적인 완성도
5. 기존 레거시 스타일 보존

기존 스타일을 유지하는 것보다, 더 통일되고 더 적절한 방향으로 정리하는 것이 우선이다.

## Non-Negotiable Redesign Policy
- 기존 UI가 있어도 필요하면 레이아웃, 간격, 타이포그래피, 색상, 컴포넌트 스타일을 과감히 수정한다.
- 이미 존재한다는 이유만으로 낡거나 어색한 디자인을 보존하지 않는다.
- 한 화면만 부분적으로 예쁘게 바꾸고 주변은 그대로 두는 식의 피상적인 수정은 하지 않는다.
- 핵심 패턴 하나를 바꿨다면, 같은 역할의 관련 컴포넌트들에도 반영해 전체적으로 맞춘다.
- "최소 수정"보다 "완성도 있는 통일감"을 우선한다. 단, 비즈니스 로직은 불필요하게 건드리지 않는다.

## Core Design Principles
- 최신 웹 UI 감각을 반영하되, 유행 요소를 장식처럼 남발하지 않는다.
- 제품 분위기에 맞는 현대적 디자인을 만든다.
- 시각적 위계가 분명해야 한다.
- 넉넉하고 정돈된 spacing을 사용한다.
- 타이포그래피는 자신감 있고 읽기 쉬워야 한다.
- 표면감, 그림자, 경계선, radius, motion은 절제되고 일관되어야 한다.
- 모바일부터 데스크톱까지 자연스럽게 반응해야 한다.
- 접근성과 실제 사용성을 희생하지 않는다.

## Consistency Rules
이 프로젝트의 UI는 하나의 디자인 언어처럼 보여야 한다.

반드시 일관되게 유지할 것:
- color palette
- typography scale
- spacing scale
- radius scale
- shadow/depth rules
- button styles
- input/form styles
- card, modal, dropdown, table styles
- icon style
- hover / focus / active / disabled / loading states
- section spacing and layout rhythm

금지:
- 페이지마다 다른 radius 사용
- 페이지마다 다른 button 철학 사용
- 컴포넌트마다 다른 shadow 스타일 사용
- 어떤 곳은 과하게 화려하고 어떤 곳은 너무 밋밋한 혼합
- 랜덤한 gradient, glass effect, border 처리 남발
- 동일 역할 컴포넌트가 서로 다른 템플릿처럼 보이는 상태
- 새 UI와 옛 UI가 나란히 보여 충돌하는 상태

## Old-Looking UI Avoidance
다음 요소는 특별한 이유가 없으면 피한다.
- 답답하고 촘촘한 레이아웃
- 지나치게 작은 텍스트
- 무거운 border 남발
- 기본 라이브러리 스타일을 거의 그대로 쓴 듯한 느낌
- 정보 위계가 약한 평면적 화면
- 간격 규칙이 들쭉날쭉한 화면
- 카드, 섹션, 버튼, 폼이 서로 다른 시대의 디자인처럼 보이는 상태
- 관리자 화면 같은 밀도와 마케팅 페이지 같은 연출이 의도 없이 섞인 상태

## Modern Quality Bar
결과물은 아래 상태를 만족해야 한다.
- 지금 시점의 제품처럼 보여야 한다
- 과하게 트렌디하지 않아도 세련되어 보여야 한다
- 프로젝트의 성격에 맞아야 한다
- 화면 간 톤과 규칙이 이어져야 한다
- "한 사람이 전체를 설계한 제품"처럼 보여야 한다

## Frontend Workflow
UI 관련 작업을 시작하면 다음 순서로 진행한다.

1. 현재 프론트엔드 구조를 먼저 파악한다.
2. 공통 레이아웃, 공통 컴포넌트, 페이지별 스타일 차이를 확인한다.
3. 어떤 부분이 낡아 보이는지, 무엇이 일관성을 깨는지 먼저 정리한다.
4. 수정 전에 목표 디자인 방향을 짧게 요약한다.
5. 가능하면 shared token / theme / reusable component부터 정리한다.
6. 화면 단위 수정이 필요하면 관련 컴포넌트까지 함께 정비한다.
7. 충돌하는 legacy 스타일은 안전하게 제거하거나 마이그레이션한다.

## Implementation Rules
- 현재 프로젝트의 프레임워크와 스타일링 방식을 우선 존중한다.
- Tailwind, CSS Modules, styled-components, Emotion, SCSS 등 기존 스택 안에서 가장 일관된 구조로 정리한다.
- 큰 마이그레이션은 명확한 필요가 있을 때만 제안한다.
- 디자인 통일을 위해 token, variable, theme, reusable primitive를 적극적으로 사용한다.
- one-off 값 하드코딩은 최소화한다.
- 비즈니스 로직, API 계약, 데이터 흐름은 꼭 필요한 범위만 수정한다.

## Behavior on UI Requests
사용자가 프론트엔드를 개선, 리디자인, 모던화, 폴리시, 정리해달라고 요청하면 아래처럼 행동한다.
- 그것을 기존 디자인을 바꿔도 된다는 허가로 해석한다.
- 단순 색상 변경이나 spacing 몇 군데 수정으로 끝내지 않는다.
- 현재 제품이 더 일관되고 더 현대적으로 보이도록 필요한 범위의 refactor를 수행한다.
- 공통 컴포넌트 수정이 필요하면 로컬 화면만 고치지 말고 공통부를 우선 정리한다.
- 결과적으로 영향을 받은 범위가 하나의 제품처럼 느껴지게 만든다.

## Definition of Done
다음 상태면 완료가 아니다.
- 한 섹션만 현대적이고 주변은 여전히 올드함
- 공통 컴포넌트 스타일이 계속 제각각임
- spacing, typography, radius, interaction 규칙이 섞여 있음
- 새로 만든 UI와 기존 UI가 서로 충돌함
- 전체적으로 패치워크처럼 보임

다음 상태여야 완료다.
- 작업 범위 안에서 디자인 언어가 통일되어 있음
- 기존보다 더 현대적이고 더 명확함
- 프로젝트 분위기와 맞음
- 낡은 느낌이 줄고 완성도가 올라감
- 화면 전체가 같은 방향성으로 정리되어 있음
